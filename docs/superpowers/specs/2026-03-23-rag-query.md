# RAG 问答 — Design Spec

> 日期：2026-03-23
> 状态：Approved (v2 — 修复 12 项 review 发现)
> 范围：P1.3 — RAG 问答系统（方案 C：上下文填充）

---

## 概述

用户在搜索页输入长问句，后端加载用户全部文章摘要塞进 LLM 上下文，LLM 综合回答并标注来源文章。无向量数据库、无嵌入管线，一次 LLM 调用搞定。

**核心洞察：** 个人知识库典型规模 50-500 篇文章，500 篇 × ~100 tokens ≈ 50K tokens，完全装进 DeepSeek 64K 上下文窗口。LLM 看到全部文章比任何检索算法都准。

## 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| RAG 方案 | 上下文填充（方案 C） | 零基础设施、最简单、LLM 直接读全部摘要最准 |
| 超过 500 篇 | FTS 预筛选 Top-50 再填充 | 降级策略，保证 context 不溢出 |
| Token 预算 | 每篇 summary 截断 150 tokens，总上限 50K tokens | 防止 context 溢出 |
| 响应模式 | 同步返回（非 SSE），后端 30s 超时 | 500 篇摘要 LLM 几秒内回答 |
| 搜索分流 | 短查询→FTS，长问句→RAG | 输入 >10 字 + 疑问词/问号 触发 RAG |
| 跟进对话 | inline thread（新回答接在下方） | 与原型一致，非重新查询 |
| 引注风格 | 上标数字 ¹²³（灰色，点击 accent） | 与原型一致 |
| 防抖 | RAG 查询 1s 防抖 | 防止重复昂贵 LLM 调用 |

## 后端设计

### 1. API 端点

#### POST /api/v1/rag/query

**请求：**
```json
{
  "question": "关于用户留存有哪些方法？",
  "conversation_id": null
}
```

**响应：**
```json
{
  "answer": "基于你的收藏，留存的本质是习惯设计...",
  "sources": [
    {
      "article_id": "uuid",
      "title": "用户留存的本质：习惯回路与变量奖赏",
      "site_name": "少数派",
      "summary": "留存不是功能问题，是行为设计问题...",
      "created_at": "2026-03-15T10:00:00Z",
      "relevance": 0.95
    }
  ],
  "source_count": 3,
  "followup_suggestions": [
    "哪些文章提到了飞轮效应？",
    "留存和增长的关系是什么？"
  ],
  "conversation_id": "uuid"
}
```

**逻辑：**
1. 验证 auth
2. 检查 Free 配额：`rag_count_this_month`，若 `rag_month_reset_at` < 本月 1 日则重置为 0。Free 限 5 次/月，超限返回 429
3. **边界检查**：用户文章数为 0 → 直接返回 `answer: "先收藏一些文章再来提问吧。"` + 空 sources，不调 LLM
4. 若有 `conversation_id`：加载历史消息（最近 10 条）作为对话上下文
5. 加载用户全部文章（id, title, summary, site_name, created_at）
6. **Token 预算控制**：每篇 summary 截断到 150 tokens（约 200 中文字符）。累计 token 数到 50K 上限时停止添加
7. 若文章数 > 500 或累计 tokens 超预算：用 pg_trgm 搜索 `question` 关键词，取 Top-50 填充
8. **输入清洗**：question 移除 ``` 和 role markers（防 prompt 注入）
9. 构建 prompt（见下方）
10. 调用 DeepSeek Chat API（temperature 0.3, max_tokens 2048, json_object response, **timeout 30s**）
11. 解析 JSON 响应：answer, cited_article_indices[], followup_suggestions[]
12. **JSON 解析失败兜底**：返回 `answer: "抱歉，回答生成失败，请重试。"` + 空 sources
13. 将 indices 映射为实际 article_id，过滤无效 index
14. 写入 rag_conversations（新建或追加）+ rag_messages
15. 递增 Free 用户 `rag_count_this_month`
16. 返回响应

**权限：** auth
**配额：** Free 5 次/月，Pro 无限

### 2. AI Prompt

**系统 prompt：**
```
你是用户的个人知识助手。以下是用户收藏的文章摘要列表。
基于且仅基于这些文章回答用户的问题。

输出 JSON 格式（不要 markdown 代码块）：
{
  "answer": "回答正文，在引用处用上标数字标注对应文章编号，如 ¹ ² ³",
  "cited_indices": [1, 3, 5],
  "followup_suggestions": ["建议追问1", "建议追问2"]
}

规则：
1. 只基于用户的收藏回答，不编造内容
2. 引用标注必须对应下方文章列表的编号
3. 回答风格：简洁、有洞察力、直击核心。对核心观点用加粗强调
4. 给出 2 个建议的跟进问题
5. 如果收藏中没有相关内容，answer 写 "你的收藏中没有找到与此相关的内容。"，cited_indices 为空
```

**用户 prompt：**
```
用户收藏（共 {count} 篇）：

[1] 《{title}》({site_name}, {date}): {summary}
[2] 《{title}》({site_name}, {date}): {summary}
...

{对话历史（如果有 conversation_id）：}
用户：{之前的问题}
助手：{之前的回答}
...

用户问题：{question}
```

## iOS 设计

### 1. 搜索分流逻辑

```swift
func isRAGQuery(_ text: String) -> Bool {
    let trimmed = text.trimmingCharacters(in: .whitespacesAndNewlines)
    guard trimmed.count > 10 else { return false }
    let questionIndicators = ["？", "?", "什么", "哪些", "如何", "为什么", "怎么", "怎样", "是否", "有没有", "能不能", "多少"]
    return questionIndicators.contains { trimmed.contains($0) }
}
```

- `isRAGQuery == false` → 现有 FTS 搜索
- `isRAGQuery == true` → 显示 RAG 回答 UI + 触发 POST /rag/query
- **防抖**：RAG 查询触发有 1s 防抖延迟，避免输入过程中多次调用

### 2. RAG 回答 UI（匹配原型 03）

**文件：** `ios/Folio/Presentation/Search/RAGAnswerView.swift`

#### 加载中状态
- "✦ 正在思考..." 文字，居中，accent 色 ✦ + textSecondary 文字
- 脉冲动画（opacity 0.4 → 1.0 循环，1.5s）

#### 回答正文
- "✦ 基于 N 篇收藏" 标签（11px, accent, uppercase, tracking 1）
- **Pull quote**（如果回答中有核心观点）：accent 左竖线 2px + 衬线粗体引文
- 正文：衬线体 16px, line-height 1.75（与原型一致，比 Reader 正文稍小）
- 行内引注：上标数字 ¹²³（12px, textTertiary 灰色，点击时 accent 色高亮）
- 点击引注 → 滚动到对应来源卡片并展开

#### 来源文章卡片
- header "来源文章"（14px, textSecondary）
- 每张卡片：
  - accent 圆点 4px + 标题（衬线 15px）+ 来源 + 日期（"3月15日收藏"）
  - 点击 → 展开原文摘录（excerpt）
  - 再次点击 → 跳转到 Reader
- 最多显示 5 张来源卡片

#### 跟进建议
- 2 个文字按钮，"→" 前缀（非 pill 样式，与原型一致）
- 点击 → **不填入搜索框**，而是在当前位置下方 inline 追加新的问答对（inline thread）

#### 跟进对话线程（follow-up thread）
- 用户的跟进问题：灰色左竖线 + textTertiary 文字
- AI 的跟进回答：和首次回答相同排版（正文 + 引注 + 来源）
- 底部：跟进输入框，placeholder "继续提问…" + 发送按钮（accent 圆形）
- 对话通过 `conversation_id` 串联，每次跟进调用 POST /rag/query 传入 conversation_id

#### 错误状态
- **LLM 超时/API 失败**："回答生成失败，请重试。" + 重试按钮
- **Free 配额用尽**："本月问答次数已用完（5/5）" + "升级 Pro，无限提问" 按钮
- **0 篇文章**："先收藏一些文章再来提问吧。" + "去首页" 按钮

#### 无相关结果
- 不是一个独立 UI 状态，而是 LLM 返回的特殊 answer
- "你的收藏中没有找到与此相关的内容。" 以正常回答样式显示
- sources 为空，不显示来源卡片区

### 3. DTO + APIClient

```swift
struct RAGQueryRequest: Codable {
    let question: String
    let conversationId: String?
}

struct RAGQueryResponse: Codable {
    let answer: String
    let sources: [RAGSource]
    let sourceCount: Int
    let followupSuggestions: [String]
    let conversationId: String
}

struct RAGSource: Codable {
    let articleId: String
    let title: String
    let siteName: String?
    let summary: String?
    let createdAt: Date
    let relevance: Double
}

// APIClient method (30s timeout):
func ragQuery(question: String, conversationId: String? = nil) async throws -> RAGQueryResponse
```

修改 HomeView 的搜索内容区：

- 保留搜索框 + 取消按钮
- RAG 回答出现在搜索结果位置
- 跟进对话在同一页面内 inline 展示（不跳转）

### 5. 搜索 ViewModel 扩展

在 SearchViewModel 或 HomeViewModel 中新增：
- `ragResponse: RAGQueryResponse?`

## 不做

- SSE 流式响应（同步足够）
- 向量嵌入
- 独立 RAG 页面（在搜索页内完成）
- 离线 RAG
- RAG 对话历史页面（跨 session 持久化）
- 保存 RAG 回答为笔记
- 打字机效果（原型有提但实际未实现，MVP 不做）
